# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A demo where a user chats a data visualization into existence in **M365 Copilot** and publishes it as a live tile on a **Power App PCF dashboard**. Natural language → a Dataverse-backed chart, rendered two ways from one generated artifact.

Components (in build order):
- **`mcp-server/`** — the actual product, and where nearly all work has happened. A Node + TypeScript MCP server (also a plain web server). **Done**: implements the full tool + web contract below, verified against the live Dataverse env.
- **`DashboardAgent/`** — an M365 Copilot **declarative agent** (Agents Toolkit / ATK project) that *consumes* the MCP server. **Functional.** Manifests only; no app logic; regenerated by ATK, so don't hand-edit lightly. The provisioned agent's **`gptId`** is what the PCF's `agentId` property must be set to — an opaque agent id, per Microsoft's time-tracker sample of the form `T_<guid>` (tenant) or `U_<guid>.declarativeAgentPowerApps` (user-scoped), i.e. **`M365_TITLE_ID`** in `env/.env.dev`, *not* `TEAMS_APP_ID`. Verify the live value by opening the agent in the Copilot pane and running `Xrm.Copilot.getCurrentAgent()` in DevTools.
- **`controls/dynamic-dashboard/` — the Power Apps PCF dashboard control (`DDB.DynamicDashboard`).** Implemented + **deployed** (`pac pcf push --publisher-prefix ddb`, bound via solution import as the default grid control of the hosting table). Full-page **dataset** PCF that fetches `GET /dashboard`, iframes each tile URL, and listens for the publish nudge. Hosting table created via MCP is **`cr19f_dashboard`** (the MCP tool uses the env default publisher prefix, not `ddb`). See "PCF component" below.
- **`spec.md`** — the authoritative design for the *server* (~14 sections). Code comments cite it as `§3`, `§6.2`, etc. When a comment says "see spec §X", `spec.md` is the source of truth. The **client-side integration** (Xrm.Copilot APIs, the widget→app postMessage bridge, the PCF's design + pitfalls) is `references/power-apps-copilot-integration.md`. `.github/agents/ddb-only.agent.md` is a scoped agent persona.

## Commands (run from `mcp-server/`)

```bash
npm install

# Fastest inner loop — host-free, no MCP client needed:
npm run probe dataverse                       # acquire token + SELECT TOP 1 (settles creds/connectivity)
npm run probe generate "<NL request>"         # full pipeline → writes dist/probe-<ts>.html to eyeball
npm run probe query "<vizId>"                 # re-run a stored viz's SQL

npm run build          # bundle the viewer to dist/mcp-app.html (Vite single-file) — REQUIRED before serve
npm run serve          # Streamable HTTP on :3101/mcp (health: /health)
npm run serve:stdio    # stdio transport instead
npm run dev            # viewer watch + server watch (concurrently)
npm run typecheck      # tsc on BOTH tsconfig.server.json and tsconfig.json
```

There is **no test runner**. Verification is manual: `probe generate` for the generation path, opening the emitted `dist/*.html` in a browser for visual/resize checks, and the live stderr trace + `logs/*.jsonl` for tool calls. `.vscode/mcp.json` already points VS Code at `http://localhost:3101/mcp`; watch the **"MCP: ddb-mcp"** output channel.

`npm run build` must run before `serve`, or the viewer resource serves a "not built" fallback page.

## Architecture (the parts that span files)

**Two planes off one Express app** (`main.ts`, one port 3101):
- **MCP plane** (`POST /mcp`, per-session `StreamableHTTPServerTransport`) — drives the Copilot chat pane. Tools + one viewer resource live in `server.ts`.
- **Web plane** (`src/web.ts`: `/dashboard`, `/tiles/:vizId`, `/tiles/:vizId/data`, `DELETE /tiles/:vizId`) — what the PCF dashboard and published tiles hit over plain HTTP.

**Generation pipeline** (`src/generate.ts`): NL description + the curated schema doc (`config/dataverse-schema.md`) → a single OpenAI **structured-output** call (`response_format: json_schema, strict`) returning `{ title, vizType, sql, renderCore }`. `create_visualization` runs a **2-attempt repair loop**: on failure it feeds the error string back into the next generation attempt. SQL is validated *by executing it* against Dataverse, not by a parser.

**"Generate once, wrap twice"** (`src/assemble-document.ts`): the LLM emits only a vanilla `function render(container, rows){…}`. `assembleDocument` wraps that core into a self-contained HTML doc in one of two data modes:
- `inline` — rows baked in; returned as `structuredContent.html` for the Copilot pane widget.
- `fetch` — the doc fetches `/tiles/:id/data?k=…` on load and on ↻; used for the served PCF tile.

The doc bundles a `DDB` helper (`window.DDB`: `fmt`, `truncate`, `palette`, `tooltip`) and re-invokes `render()` on resize so charts **reflow** rather than scale. Theme comes in via `postMessage({type:'set-theme'})`; height goes out via `postMessage({type:'widget-resize'})`.

**`vizType` is soft metadata, not an engine constraint.** `assembleDocument` never branches on it — it just runs whatever `render()` the LLM wrote. The `bar|line|pie|kpi` set is enforced only by (1) the enum in the structured-output schema and (2) the "DESIGN BRIEF" in `generate.ts`'s `systemPrompt()`. To add chart types or multi-series charts, change the brief + schema doc conventions (and optionally widen the enum) — no rendering-engine change is needed.

**Rendering is vanilla-only, on purpose.** No CDNs, no charting libs, no `fetch/eval/import` inside the widget — so the output is immune to host CSP variance. `src/security.ts` enforces this: `sanitizeRenderCore` rejects external URLs (allowlisting only W3C SVG/XML namespace URIs) and forbidden tokens; `sanitizeSql` allows a single read-only `SELECT`/`WITH`. Both throw, and the repair loop catches the throw. If you change the DESIGN BRIEF to emit new constructs, check them against these sanitizers first.

**Data layer** (`src/dataverse.ts`): confidential-client (`ClientSecretCredential`) token for `${orgUrl}/.default`, attached to an `mssql` pool against the Dataverse **TDS/SQL endpoint** (`authentication.type = 'azure-active-directory-access-token'`, cast through `sql.config` because it's outside mssql's TS types). Read-only. `DATA_PATH=webapi` (OData) is stubbed and throws — `tds` is the implemented path.

**State** (`src/store.ts`): a JSON-file-persisted singleton (`state.json`, gitignored). `vizStore` holds every generated viz + its bound SQL/renderCore; `registry` is the dashboard (vizIds the user clicked "Add to dashboard"). `add_to_dashboard` is idempotent.

**Capability tokens** (`src/security.ts` `tokenOk`, constant-time): `DASHBOARD_KEY` (long-lived, gates `/dashboard` + `DELETE`) and a per-tile `tileKey` minted at add-time (gates that tile's page + data). Tiles are framed by Power Apps, so `/tiles/:vizId` sets `Content-Security-Policy: frame-ancestors https://*.dynamics.com …`.

**The publish nudge** (`src/mcp-app.ts`): the viewer's "Add to dashboard" button calls the `add_to_dashboard` server tool, then fans a `postMessage` (`eventName: 'powerapps.copilot.chat.action'`, `action: 'ddb.dashboard.updated'`) to `window.top/parent/parent.parent` so the host Power App learns a new tile exists.

## PCF component

The dashboard control. Its binding contract is the server's **web plane** — develop against the running server (no mock needed).

**Endpoints it consumes** (shapes are exact; from `spec.md` §7): `GET /dashboard?k=<dashboardKey>` → `{ tiles: [{ vizId, title, vizType, tileUrl, addedAt }] }`; `GET /tiles/:vizId?k=<tileKey>` → self-contained, self-refreshing tile HTML; `GET /tiles/:vizId/data?k=<tileKey>` → `{ rows }`; `DELETE /tiles/:vizId?k=<dashboardKey>` → `204` (idempotent).

**Design** (`references/power-apps-copilot-integration.md` §4): a standard **non-React dataset** PCF `DDB.DynamicDashboard`, hosted full-page as the default grid control of the empty `cr19f_dashboard` hosting table (the dataset is ignored — it's just a host surface; **never a custom page**, which has no `window.Xrm`). Manifest input props: `serverBaseUrl`, `dashboardKey`, `agentId` (= the gptId above), `seedPrompt`, `autoRefreshSeconds`; `external-service-usage` domain = the server host. No `WebAPI` (writes nothing to Dataverse).

**Key behaviors:** `loadDashboard()` fetches and **diffs by `vizId`** (recreating unchanged iframes reloads every chart — visibly ugly); tiles carry `updatedAt`, and a changed stamp reloads just that tile's iframe (in-place edits via `update_visualization`). Each tile is `<iframe src="${tileUrl}" sandbox="allow-scripts allow-same-origin" loading="lazy">` — `allow-same-origin` is **required** or the tile's self-`fetch` of its `/data` becomes a null-origin CORS failure. "Generate new report ✨" = `window.Xrm.Copilot.openM365CopilotPanel()` + `sendPromptToM365Copilot(prompt, { autoSubmit:true, gptId })` with a **random prompt from the baked-in `SEED_PROMPTS` list** (each drawn from the generator's curated schema so it reliably produces a valid viz). **Nudge receive = dual registration** (belt-and-suspenders, because the pane is a nested iframe): `Xrm.Copilot.addActionHandler('ddb.dashboard.updated', h)` **and** a raw `window.addEventListener('message', …)` filtering `eventName==='powerapps.copilot.chat.action' && action==='ddb.dashboard.updated'`; treat it only as a refresh trigger and reconcile via `/dashboard` (never render solely from `actionData`).

**Critical config coupling — HTTPS:** the `tileUrl`s the server returns are built from `PUBLIC_BASE_URL` (`mcp-server/.env`). The PCF runs inside an **HTTPS** Power App, so an `http://localhost` server URL is silently blocked as mixed content. Run the server behind a **devtunnel** (already installed) and set **both** `PUBLIC_BASE_URL` and the PCF's `serverBaseUrl` to that HTTPS base (e.g. `https://<id>-3101.euw.devtunnels.ms`). The manifest `external-service-usage` domain must match; changing it = manifest edit + **version bump + `pac pcf push`**.

**Toolchain (verified present):** `pac` 2.8.x, dotnet 10 SDK, Node 24. Build with `npm run build` in the control dir; deploy with `pac pcf push --publisher-prefix ddb` (**bump the ControlManifest version every push** or UCI serves a stale bundle) + control binding on `cr19f_dashboard` + a **Dashboard** sitemap page.

## Conventions

- **ESM throughout** (`"type":"module"`). Relative imports use `.js` extensions even from `.ts` sources.
- **Two tsconfigs**: `tsconfig.server.json` (Node; excludes the DOM viewer) and `tsconfig.json` (DOM; for `src/mcp-app.ts`). `typecheck` runs both.
- **Logging** (`src/logger.ts`): `redact()` scrubs secret-like keys; `startCallLog(tool, runId)` writes per-call JSONL + a stderr mirror; errors returned to the client carry a short `(log: <runId8>)` handle. Never log raw secrets — use `configSummary()` for a redacted config view.
- **Config** (`src/config.ts`): all env access is centralized here; `requireConfig([...])` throws a clear message for missing keys. `mcp-server/.env` is gitignored and holds **real** OpenAI + AAD secrets — do not echo, log, or commit it.
